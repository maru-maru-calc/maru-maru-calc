# Landing operation video rules

LP の四則演算デモ動画を作るときのルール。

## 共通ルール

- 出力サイズは `640x360` に統一する。
- LP の動画枠は `16:9` 前提なので、別サイズで書き出さない。
- 録画時のゲーム内部フィールド高さは `fieldHeight=360` になる条件を使う。
- `fieldHeight` が変わるとボウルの底のカーブ形状が変わるため、動画間で見た目が揃わない。
- 画角はボウルの底のカーブ、操作対象、結果のまるが自然に見える位置にする。
- 背景装飾の泡、演算ボタン、式表示など、説明に不要なUIは録画時だけ非表示にする。
- 関係ない泡やまるが画面に残らないようにする。
- 花火やクリア演出は入れない。演算理解のノイズになるため。
- 最終状態は短すぎず長すぎず、約1秒前後見える程度にする。
- 他の動画とテンポが揃うように、基本は実時間に近いテンポで書き出す。

## かける動画のルール

かける動画は、次の流れが見える必要がある。

1. 元の泡を割る。
2. 中のまるがボウルに落ちる。
3. `×` を選ぶ。
4. 対象のまるが再度あわに包まれる。
5. `×3` などの倍率の泡を選ぶ。
6. あわが増えて、かけ算の過程が見える。
7. あわが弾ける。
8. 最終的なまるの数が見える。

避けること:

- 最初の泡が割れずにそのまま下へ落ちる見え方にしない。
- `×3` の泡だけが先に見えて、元のまるがどうできたか分からない構成にしない。
- あわが弾ける前に動画を終えない。
- 最終状態を長く保持しすぎて、次のループまで間延びさせない。
- 関係ない泡を残さない。

現在のかける動画は、`4` の泡を割ってまるを落とし、その後 `×` で再度あわに包み、`×3` で増える流れを使っている。

## 確認観点

動画を差し替えたら、少なくとも次を確認する。

- 他の四則演算動画と出力サイズが同じか。
- ボウルの底の形が他の動画と揃っているか。
- 説明に関係ない泡、魚、UI、花火が映っていないか。
- 操作の過程が読み取れるか。
- 最終状態が短すぎないか、長すぎないか。
- ループに戻るタイミングが自然か。

## 回帰確認

LP 動画差し替え後は、最低限次の E2E を実行する。

```sh
npm run e2e -- e2e/app-design-flow.spec.ts -g "landing page presents the core copy and opens the app|landing page clicks do not start background music" --workers=1
```
